## v1.0 – 基础透传（截图+指令）
**目标**：跑通最小闭环，建立性能基线。

**技术细节**（同你写的，略调整）：

+ 客户端：Qt `QTcpSocket` 发送 JPEG（`QBuffer` + `QDataStream`）。
+ 网关服务：MyMuduo（Linux），维护 session 映射（内存 `std::unordered_map`）。
+ 服务间通信：Unix Domain Socket（`AF_UNIX`），协议 `[len:4B][type:1B][data]`。
+ 截图：`QScreen::grabWindow` → JPEG 质量 70%，固定 5 fps。
+ 指令：鼠标/键盘事件 → JSON（`{"type":"mouse_move","x":100,"y":200}`）。

**测试环境**：三个服务跑同一台 Linux（虚拟机/WSL2），被控端/控制端 Windows 物理机，同一局域网。

**性能指标（基线）**：

| 指标 | 目标值 | 测量方法 |
| --- | --- | --- |
| 端到端延迟 | < 250 ms | 截图中嵌入时间戳（例如第一个像素点存储时间戳），控制端解码后读取 |
| 带宽 | < 8 MB/s | `nethogs` 或 `tcpdump` 统计 |
| CPU（被控端） | < 40% | Windows 任务管理器 |
| CPU（gateway） | < 15% | `perf top -p pid` |
| 最大会话数 | 10 | 逐步增加客户端，直至延迟 > 500ms |


**不足与风险**：

+ 无变化检测，带宽浪费。
+ Unix Socket 数据拷贝多，内部转发延迟高（~100 µs）。
+ 无加密，会话信息明文。
+ 无断线重连，服务重启全部掉线。

**实验笔记要点**：

+ 画出完整数据流图：被控端 → gateway → screen_stream → 控制端；控制端 → gateway → control_signal → 被控端。
+ 记录第一次延迟分布（可画直方图，横轴延迟 ms，纵轴次数）。
+ 指出瓶颈：截图压缩（CPU 70%）还是网络（带宽）？通常 JPEG 压缩是最大瓶颈。

**这个版本你会学到什么（八股文 + 代码冲击）**：

1. **进程间通信（IPC）基础**：
    - 什么是 IPC？为什么服务间需要通信？
    - IPC 有哪些方式？（管道、FIFO、信号、消息队列、共享内存、Unix Socket、网络 Socket）
    - 我们为什么选 **Unix Domain Socket**？它和 TCP Socket 的区别（不需要 IP + 端口，不走网络协议栈，更快）。
2. **TCP 粘包与协议设计**：
    - 为什么需要自定义协议？`[len][type][data]` 如何解决粘包？
    - 大端/小端问题（网络字节序转换）。
3. **代码冲击**：
    - 第一次看到多个独立进程（gateway、screen_stream、control_signal）协作运行。
    - 看到 `socketpair` 或 `bind` 到 `@abstract` 或 `/tmp/.socket` 的 Unix Socket 文件。

## v2.0 – Redis 会话管理（从单机内存到集中缓存）
**目标**：将会话状态从 gateway 内存迁移到 Redis，实现会话持久化、设备心跳、跨服务通知。

**技术细节**：

+ 使用 hiredis 或 redis++。
+ 数据结构：
    - `device:{device_id}` → Hash（`ip`， `port`， `online`， `last_seen`）。
    - `session:{session_id}` → Hash（`device_id`， `control_client_id`， `create_time`）。
+ 心跳：被控端每 10 秒发送 PING → gateway 更新 Redis 中 `last_seen`，并设置 key 过期时间 30 秒。
+ Pub/Sub：当设备离线时，gateway 发布 `device_offline:{device_id}` 事件，其他服务（如 control_signal）订阅并清理本地缓存。

**测试环境**：增加 Redis 容器（`docker run -p 6379:6379 redis`）。

**性能指标（新增）**：

| 指标 | 目标 | 测量方法 |
| --- | --- | --- |
| 注册延迟 | < 10 ms | 记录 Redis `HSET` 耗时（平均） |
| 会话恢复时间（服务重启） | < 500 ms | 重启 gateway 后，客户端重连到画面出现的时间 |
| Redis QPS | < 1000 | `redis-cli info stats` |


**不足与风险**：

+ Redis 单点（后续可加 Sentinel，但本版本不做）。
+ 无持久化配置（可开启 AOF，但会增加延迟，暂不开）。

**这个版本你会学到什么**：

1. **Redis 核心数据结构与适用场景**：
    - 为什么这里用 Hash？而不是 String 或 Set？
    - Hash 适合存储对象的多个字段（device_id 的多个属性）。
    - 什么时候用 String？比如计数器、简单 KV。什么时候用 List？消息队列。什么时候用 Set？标签系统。什么时候用 Sorted Set？排行榜、延迟队列。
    - **对比 MySQL**：MySQL 也能存储会话，但 Redis 是内存数据库，读写延迟低（微秒级），适合高频读写、过期自动删除的场景。
2. **缓存设计模式**：
    - 旁路缓存（Cache-Aside）：先查 Redis，没有再查数据库（本版本没有数据库，但可引申）。
    - 过期时间与主动失效（心跳保活 + TTL）。
    - 发布订阅（Pub/Sub）用于跨服务事件通知。
3. **代码冲击**：
    - 第一次在 C++ 项目中集成第三方中间件（Redis）。
    - 看到服务重启后，客户端重连能恢复会话（从 Redis 读取 session）。

## v3.0 – 共享内存无锁队列（服务间通信极致优化）
**目标**：将 gateway ↔ screen_stream 之间的通信从 Unix Socket 替换为共享内存 + 无锁环形队列，降低延迟和拷贝。

**技术细节**：

+ 使用 POSIX `shm_open` + `mmap`。
+ 实现 SPSC（单生产者单消费者）环形队列：
    - 内存布局：`[read_pos:8B][write_pos:8B][data_region]`，对齐到 64 字节避免伪共享。
    - 使用 `std::atomic<size_t>` 和 `memory_order_release/acquire`。
    - 写入时先检查可用空间，然后 `memcpy`，最后更新 `write_pos`。
+ 队列大小固定（例如 16MB，存储若干帧）。
+ 当队列满时，策略：丢弃最老的一帧（覆盖）。

**测试环境**：同一台 Linux 机器，三个进程。对比 v2.0 测量内部转发延迟（在共享内存中放时间戳，消费者读出后计算差值）。

**性能对比**：

| 指标 | v2.0 (Unix Socket) | v3.0 (共享内存) |
| --- | --- | --- |
| 内部转发延迟（P99） | ~100 µs | <5 µs |
| CPU 系统占用 | 15% | 8% |


**不足与风险**：

+ 队列容量固定，满时丢帧（可接受，实时性优先）。
+ 仅单机有效，跨机器无法使用。
+ 需要手动管理共享内存创建和销毁（`shm_unlink`）。

**这个版本你会学到什么**：

1. **进程间通信进阶：共享内存**：
    - 共享内存是 IPC 中最快的方式，因为不需要内核态/用户态数据拷贝（但仍有 `memcpy`，后续 v5.0 零拷贝可消除）。
    - 共享内存本身不提供同步，需要配合锁或原子操作。这里我们用了无锁队列。
2. **无锁编程基础**：
    - 什么是原子操作？`std::atomic` 的 `load`/`store` 默认使用 `memory_order_seq_cst`（最严格），我们为什么可以用 `release/acquire`？
    - 伪共享（False Sharing）是什么？为什么要把 `read_pos` 和 `write_pos` 放在不同缓存行？
    - 环形队列比链表好在哪里（避免动态内存分配，缓存友好）。
3. **代码冲击**：
    - 第一次看到多个进程通过 `mmap` 映射同一块物理内存。
    - 看到 `memcpy` 字节流从 gateway 进程直接写入共享内存，screen_stream 进程瞬间读取到。
    - 看到无锁的 `while (!compare_exchange_weak)` 自旋。

## v4.0 – Kafka 异步审计 + RocksDB 本地缓冲
**目标**：实现可靠审计日志（所有截图帧采样 + 控制指令），并保证消息不丢失。

**技术细节**：

+ **Kafka**：使用 `librdkafka`。
    - Topic: `screen_record`（采样，每 10 帧发一次），`audit_log`（所有指令）。
    - 生产者配置 `acks=all`，`retries=3`。
+ **RocksDB**：嵌入到 gateway 进程（或独立 storage_server）。
    - 当 Kafka 发送失败时，将消息写入 RocksDB（key=自增 id，value=消息体）。
    - 后台线程定期从 RocksDB 读取并重试发送到 Kafka，成功则删除。
+ 使用 `spdlog` 进行结构化日志输出，可同时写入 Kafka 和本地文件。

**测试环境**：Docker Compose 启动 Kafka + Zookeeper（单节点）。RocksDB 嵌入 gateway。

**性能指标**：

| 指标 | 目标 |
| --- | --- |
| 异步写入额外 CPU 开销 | < 5% |
| 消息可靠性（Kafka 宕机 1 分钟） | 恢复后 0 丢失 |
| RocksDB 写入延迟（P99） | < 1 ms |


**不足与风险**：

+ Kafka 单节点（生产环境需集群）。
+ RocksDB 可能出现写放大，但本场景写入量不大。

**这个版本你会学到什么**：

1. **消息队列（Kafka）核心概念**：
    - 为什么用消息队列？异步解耦、削峰填谷、持久化。
    - Topic、Partition、Offset、Consumer Group 是什么？
    - 如何保证消息不丢失？生产者 `acks=all` + 消费者手动提交 offset。
    - 对比其他 MQ（RabbitMQ 适合低延迟，Kafka 适合高吞吐）。
2. **嵌入式存储引擎（RocksDB）**：
    - LSM 树 vs B+ 树。RocksDB 写性能好，读可能稍慢（适合写多读少场景）。
    - 如何作为本地队列使用？保证 Kafka 挂了数据不丢。
    - `WriteBatch` 批量写入提高性能。
3. **代码冲击**：
    - 第一次看到异步回调：Kafka 生产者发送后立刻返回，后台线程处理结果。
    - 看到 RocksDB 的 `Put`、`Get`、`Delete` 操作。
    - 看到消息的“可靠队列”实现：写入 RocksDB → 后台重试 → 成功后删除。

## v5.0 – io_uring + 零拷贝（网络层优化）
**目标**：替换 gateway 的网络事件循环（从 epoll 到 io_uring），并实现零拷贝发送屏幕帧。

**技术细节**：

+ 使用 `liburing`，初始化 `io_uring` 队列（大小 256）。
+ 替换 Muduo 的 `EventLoop` 为 `IoUringLoop`：提交 `IORING_OP_ACCEPT`、`IORING_OP_RECV`、`IORING_OP_SEND`。
+ **零拷贝发送**：对于来自共享内存的帧数据，使用 `splice()` 将数据从共享内存文件描述符直接发送到 socket。
    - 前提：共享内存是通过 `memfd_create` 或普通文件 `mmap` 创建的，具有文件描述符。
    - 步骤：创建 pipe → `splice(shm_fd, NULL, pipe_fd, NULL, len, SPLICE_F_MOVE)` → `splice(pipe_fd, NULL, socket_fd, NULL, len, 0)`。或者直接使用 `sendfile` 如果共享内存是文件。

**测试环境**：Linux 内核 5.10+（推荐 6.x）。对比 v4.0 测量高并发（200+ 连接）下的延迟和系统调用次数。

**性能对比**：

| 指标 | v4.0 (epoll+拷贝) | v5.0 (io_uring+零拷贝) |
| --- | --- | --- |
| P99 网络延迟（200 conn） | 15 ms | 10 ms |
| 系统调用次数/秒 | 200k | 40k |
| CPU 系统态 | 15% | 6% |


**不足与风险**：

+ 内核版本要求高（5.10+），部分云服务器可能较旧。
+ `splice` 零拷贝在某些文件系统/网卡驱动下可能回退，需加 fallback 逻辑。

**这个版本你会学到什么**：

1. **io_uring 与 epoll 对比**：
    - epoll 是事件通知机制，仍需系统调用（`epoll_wait`）后执行 `read`/`write`。
    - io_uring 是提交队列（SQ）和完成队列（CQ），批量提交，减少系统调用次数。
    - 适用场景：高 IOPS、大量读写，比如数据库、代理服务器。
2. **零拷贝技术**：
    - 传统 `read` + `write` 需要 4 次上下文切换 + 2 次拷贝。
    - `sendfile` 或 `splice` 可在内核态完成拷贝，减少到 2 次上下文切换 + 0 次用户态拷贝。
    - 共享内存本身已经减少了一次内核→用户拷贝，加上 `splice` 可达真正零拷贝。
3. **代码冲击**：
    - 第一次看到 `io_uring_enter` 系统调用，以及 `sqe`（提交队列项）的构造。
    - 看到 `splice` 如何将文件数据直接“灌”到 socket。
    - 看到 P99 延迟的显著改善和系统调用次数的断崖式下降。

## v6.0 – C++20 协程改造 + TLS 加密（可选，但强烈推荐）
**目标**：提高异步代码可读性，并为通信安全增加 TLS 加密。

**技术细节（协程）**：

+ 在 gateway 中，将 `onMessage` 回调改为协程：`co_await async_read()`，然后 `co_await async_write()`。
+ 实现简单的 `Awaiter` 包装 `io_uring` 的完成事件。
+ 仅改造控制指令处理路径，画面数据路径可不动（性能敏感）。

**技术细节（TLS）**：

+ 使用 OpenSSL 1.1.1+，为 gateway 增加 TLS 支持（`SSL_CTX`，`SSL_read`/`SSL_write`）。
+ 客户端使用 `QSslSocket`（Qt 内置）。
+ 测试环境用自签名证书。

**性能影响**：

+ 协程：代码可读性提升，性能无明显变化。
+ TLS：连接建立增加约 30ms（握手），数据加解密增加 CPU 约 10%。

**不足与风险**：

+ 协程需要编译器支持（GCC 11+ 或 Clang 15+），且调试不便。
+ TLS 证书管理复杂（可后续自动化）。

**这个版本你会学到什么**：

1. **C++20 协程**：
    - 协程是什么？与线程的区别（协程是用户态调度，没有上下文切换开销）。
    - `co_await`、`co_return`、`co_yield` 的作用。
    - 如何实现自己的 `Awaiter`？`await_ready`、`await_suspend`、`await_resume`。
2. **网络安全基础（TLS）**：
    - 为什么需要 TLS？防止窃听、篡改、中间人攻击。
    - TLS 握手过程（ClientHello、ServerHello、证书验证、密钥交换）。
    - 与 SSL 的关系。TLS 1.3 相比 1.2 的改进（减少一次 RTT）。
3. **代码冲击**：
    - 第一次看到函数可以 `co_await` 一个异步操作，代码像同步一样顺序执行。
    - 看到 `SSL_read` 和普通 `read` 的区别，以及 `BIO` 的使用。

## v7.0 – MySQL 用户系统 + Docker 部署 + 最终压测
**目标**：补全完整后端能力，支持用户注册登录、设备绑定，并实现一键部署。

**技术细节**：

+ **用户系统**：新增 `user_server` 服务，处理注册/登录。
    - MySQL 表：`users(id, username, password_hash, created_at)`，`devices(device_id, user_id, device_name, last_seen)`。
    - 密码使用 bcrypt 哈希（或 `libsodium`）。
    - 登录成功后返回 JWT（JSON Web Token），后续请求携带 token。
+ **JWT 验证**：gateway 验证 token 有效性，提取 `user_id`。
+ **部署**：
    - 每个服务（gateway, screen_stream, control_signal, user_server）都打包成 Docker 镜像。
    - 编写 `docker-compose.yml` 编排所有服务 + Redis + Kafka + MySQL。
    - 提供一键启动脚本 `./start.sh`。
+ **最终压测**：使用脚本模拟 50 个控制端同时连接，测量端到端延迟、CPU、内存。

**性能指标（新增）**：

| 指标 | 目标 |
| --- | --- |
| 注册/登录延迟 | < 100 ms |
| JWT 验证开销 | < 1 ms |
| 部署启动时间 | < 5 分钟 |


**不足与风险**：

+ MySQL 单点（可后续加主从）。
+ 没有实现用户权限分级（例如管理员查看所有设备）。

**这个版本你会学到什么**：

1. **关系型数据库（MySQL）设计**：
    - 何时使用 MySQL？需要 ACID、复杂查询、数据一致性。
    - 与 Redis 的对比：MySQL 持久化、支持 SQL，但速度慢；Redis 快但功能弱。
    - 索引设计：`username` 唯一索引，`user_id` 外键。
2. **JWT 与用户鉴权**：
    - JWT 的结构：Header、Payload、Signature。
    - 为什么 JWT 适合分布式系统（无需服务端存储 session）。
    - JWT 的缺点（无法主动失效，除非加黑名单）。
3. **容器化部署（Docker + Compose）**：
    - Dockerfile 编写（多阶段构建，减小镜像体积）。
    - docker-compose 网络配置、依赖顺序（`depends_on`）。
    - 环境变量注入配置。
4. **最终压测报告**：
    - 汇总 v1.0 到 v7.0 的性能演进曲线。
    - 分析每个优化点的实际收益（例如共享内存减少延迟 98%）。
