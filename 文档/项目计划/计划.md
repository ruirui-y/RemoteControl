# 远程控制软件 —— 底层技术进阶路线（修订版）
架构：控制端(Qt) ↔ Linux中转(MyMuduo) ↔ 被控端(Qt)  
协议：自定义二进制协议，包含 TotalLen + HeaderLen + Header + Body  
总时长预估：业余时间 3~6 个月

---

## v1.0 – 基础中转透传（TCP 双通道）
### 目标
+ 实现最小可行系统：被控端注册、控制端请求连接、中转建立会话。
+ 被控端定时截屏 → JPEG → 通过数据通道发送到中转 → 中转转发给控制端。
+ 控制端通过控制通道发送鼠标/键盘指令 → 中转转发给被控端。
+ 两条独立 TCP 连接（数据端口 + 控制端口），避免队头阻塞。

### 技术栈
| 组件 | 方案 |
| --- | --- |
| 控制端/被控端 | Qt 5/6 + MyMuduo 的 `TcpClient`<br/>（移植到 Windows？需要 muduo 支持 Windows？实际上 muduo 是 Linux-only，但你可以用 Qt 自己的 `QTcpSocket`<br/> 实现客户端，避免依赖 muduo 客户端。明确：客户端不使用 MyMuduo，而是用 Qt 原生网络，因为 MyMuduo 依赖 epoll 无法在 Windows 编译。重新修正：客户端用 Qt `QTcpSocket`<br/>；中转服务器用 MyMuduo（Linux）。 |
| 中转服务器 | Linux + MyMuduo (epoll, Reactor) |
| 协议 | 自定义二进制协议，解析器在客户端和中转都要实现（Qt 端用 `QDataStream`<br/> 或手动字节操作） |
| 截屏 | Qt：`QScreen::grabWindow`<br/> → `QPixmap`<br/> → `save()`<br/> 为 JPEG（QBuffer 转字节数组） |
| 指令 | 控制端发送 JSON（如 `{"type":"mouse_move","x":100,"y":200}`<br/>）放入 Body |


### 局限
+ 中转单进程单线程，所有数据透传，无任何应用层优化。
+ 截屏定时拉取（固定 5 fps），无变化检测，浪费带宽。
+ 会话管理仅内存 map，中转重启则全部掉线。
+ 无流量控制（发送方过快可能撑爆接收方缓冲区）。
+ 无鉴权、无加密。
+ 客户端使用 `QTcpSocket`，需要自己处理分包（因为 TCP 流式）。

### 性能指标（基线，1080p，5 fps）
| 指标 | 预期值 |
| --- | --- |
| 端到端延迟（B截屏 → A显示） | 150~250 ms（含两次中转转发、JPEG 编解码） |
| 带宽 | 5~8 MB/s |
| CPU（中转） | 很低（仅 memcpy 转发） |
| CPU（被控端） | 30~40%（截图+压缩） |
| 最大并发会话 | 受中转文件描述符限制（默认 1024） |


### 测试环境
+ 中转：Ubuntu 22.04 虚拟机（或本机 WSL2），分配 2 核 4GB。
+ 被控端：Windows 物理机，Qt 程序。
+ 控制端：同一台 Windows 或另一台虚拟机。
+ 网络：同一局域网（模拟客户端在局域网，中转在同一交换机）。  
不需要云服务器。

### 协议实现要点（v1.0 即落地）
+ 客户端与中转之间所有数据包都按 `TotalLen + HeaderLen + Header + Body` 封装。
+ `PacketHeader` 至少包含：`cmd`（1字节，如 0x01=画面数据，0x02=控制指令）、`session_id`（8字节）。
+ 接收方先读 4 字节得到 `TotalLen`，再读 `TotalLen-4` 字节得到完整包，然后解析 `HeaderLen` 并拆出 Header 和 Body。

## v2.0 – 增加变化区域检测（减少带宽）
### 目标
+ 被控端不再传输整张 JPEG，而是检测屏幕变化区域，只发送变化的矩形块。
+ 大幅降低带宽占用，尤其静止画面时几乎不发送数据。

### 技术栈（新增/变更）
+ 帧间差分：缓存上一帧 RGB 数据，逐块比较（例如 16x16 块）。
+ 压缩变化块：将多个变化矩形合并后，使用 libjpeg-turbo 压缩。
+ 协议扩展：Body 中包含矩形个数、坐标、长度等。

### 局限
+ 变化检测本身消耗 CPU（逐块比较）。
+ 快速滚动场景下，变化区域可能接近全屏，带宽优势消失。
+ 实现复杂度增加，需处理矩形合并、碎片发送。

### 性能指标（对比 v1.0，静止桌面场景）
| 指标 | v1.0 | v2.0 | 变化 |
| --- | --- | --- | --- |
| 带宽（静止） | 5~8 MB/s | <0.1 MB/s（只发少量变化） | -98% |
| 带宽（全屏滚动） | 5~8 MB/s | 相近（略高，因矩形头部开销） | ±10% |
| CPU（被控端） | 30% | 45%（增加差分） | +50% |
| 延迟 | 150~250ms | 150~250ms | 不变 |


### 测试环境
与 v1.0 相同。

## v3.0 – 引入 Redis 会话管理（为中继集群准备）
### 目标
+ 将会话状态从中转内存迁移到 Redis，支持中转重启后会话不丢失。
+ 为后续多中转实例（负载均衡）做准备。

### 技术栈（新增）
+ Redis（hiredis 客户端库，集成到中转服务器）
+ 心跳机制：被控端定期发送 ping，中转更新 Redis 中的 TTL（如 30 秒）。

### 局限
+ 仍为单中转实例，但状态持久化。
+ Redis 成为新依赖，需额外维护。
+ 没有解决单点故障（中转挂了，即使会话存在，也无法转发）。

### 性能指标（新增）
| 指标 | 预期 |
| --- | --- |
| 连接建立延迟（含 Redis 写/读） | +5~10 ms（相比纯内存） |
| Redis QPS | 很低（仅注册、心跳、查询） |
| 会话恢复时间（中转重启） | 秒级（需客户端重连并重新绑定） |


### 测试环境
+ 增加 Redis 容器（Docker）或本地安装。
+ 同一台 Linux 机器跑中转 + Redis。

## v4.0 – 微服务拆分 + 共享内存无锁队列（单机高性能）
### 目标
+ 将中转服务器拆分为三个独立进程（同一台机器）：
    - `gateway`：处理客户端 TCP 连接，收发原始数据包。
    - `screen_stream`：处理画面数据的存储与转发（可能做缩放、多播）。
    - `control_signal`：处理控制指令路由。
+ 服务间通信使用 共享内存无锁队列（SPSC/MPSC），替代 localhost TCP，降低延迟和拷贝。

### 技术栈（新增/变更）
+ boost::interprocess 或 POSIX `shm_open` + `mmap`
+ 手写环形队列（基于原子变量），缓存行对齐（`alignas(64)`）
+ 进程间同步：`std::atomic` + `memory_order_acquire/release`

### 局限
+ 所有服务仍在一台机器，不能水平扩展。
+ 共享内存大小固定，需要背压策略（队列满时丢弃画面帧）。
+ 服务间无自动发现（硬编码共享内存名称）。

### 性能指标（对比 v3.0，内部转发延迟）
| 指标 | v3.0（TCP loopback） | v4.0（共享内存） | 变化 |
| --- | --- | --- | --- |
| 帧从 gateway 到 screen_stream 延迟 | ~100 µs | ~2 µs | -98% |
| 拷贝次数 | 2 次（用户态↔内核） | 1 次（直接写 shm） | -50% |
| 吞吐上限 | 受 socket buffer 限制 | 受内存带宽限制 | 更高 |


### 测试环境
+ 同一台 Linux 机器（多核），至少 4 核。
+ 压测：模拟 100 个控制端同时接收画面。

## v5.0 – io_uring + 零拷贝 + C++20 协程
### 目标
+ 网络层从 epoll 迁移到 io_uring，降低系统调用开销。
+ 屏幕帧从共享内存直接发送到网卡（零拷贝），使用 `splice()` 或 `sendfile()`。
+ 控制指令处理流程用 C++20 协程 重写，提高可读性。
+ 微观优化：分支预测、内存预取。

### 技术栈（新增/变更）
+ liburing（内核 5.10+）
+ `splice()` / `sendfile()` 配合 io_uring
+ C++20 协程（GCC 11+）
+ `__builtin_expect`、`__builtin_prefetch`

### 局限
+ 需要 Linux 内核 ≥ 5.10（推荐 6.x）。
+ 协程调试困难，编译可能遇到问题。
+ 零拷贝依赖文件描述符类型（普通文件支持，socket 需要特定内核版本）。

### 性能指标（对比 v4.0）
| 指标 | v4.0 | v5.0 | 变化 |
| --- | --- | --- | --- |
| P99 延迟（高并发 200+ 会话） | 可能有毛刺 | 更平滑 | -30% |
| CPU 系统态占用 | 15% | <8% | -50% |
| 吞吐量（多客户端） | 基准 | +40% | 提升 |


### 测试环境
+ Linux 内核 ≥ 6.0（推荐 Ubuntu 22.04 升级内核或使用 24.04）。
+ 其他同 v4.0。

## v6.0 – QUIC / WebRTC 数据通道（弱网优化）
### 目标
+ 在弱网（高丢包、高延迟）下保持可用性。
+ 控制通道使用 QUIC（0-RTT、连接迁移）。
+ 屏幕帧传输使用 WebRTC DataChannel 或 QUIC 不可靠流。

### 技术栈（新增）
+ msquic 或 google/quiche 库（C/C++）
+ libdatachannel（WebRTC）
+ 自适应码率：根据丢包率动态调整 JPEG 质量/帧率

### 局限
+ 复杂度极高，库集成可能耗时。
+ 需要处理 UDP NAT 穿透（STUN/TURN）。
+ 与现有 TCP 架构并存，代码维护量大。

### 性能指标（弱网下对比 v5.0 TCP）
| 场景 | v5.0（TCP） | v6.0（QUIC/WebRTC） | 效果 |
| --- | --- | --- | --- |
| 丢包 5% | 画面卡顿，延迟飙升 | 画面略微模糊，延迟平稳 | 体验极大改善 |
| 连接迁移 | 断线重连 | 无感知 | 本质提升 |


### 测试环境
+ 模拟弱网：`tc netem` 增加丢包/延迟。
+ 可选云服务器验证公网效果。
